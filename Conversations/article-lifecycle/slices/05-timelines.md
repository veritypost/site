# Slice 05 — Timelines

**Status:** locked
**Locked:** 2026-04-29
**Session:** 6 (investigation + Q&A)

---

## What this slice covers

How editorial timelines are authored, stored, and rendered — and the structural redesign that turns the product from a flat collection of articles into a stories-as-containers model. A "timeline" in this slice means the dated event sequence that scaffolds a story, not the user activity feed or kids reading log.

This slice introduced the biggest architectural change in the program: stories become the top-level navigable entity, articles live inside stories, and the timeline is the navigation spine that connects both plain events and article entries under one slug.

---

## How timelines work today

**What "timeline" means.** Each article row in the DB has a child set of `timelines` rows — typically 4–10 dated events generated by the LLM during article creation. They're meant to scaffold the news: "how we got here" as a vertical list of dates and labels.

**DB schema.** `timelines` table (`database.ts:9630`): `id`, `article_id` (FK to articles), `event_date` (timestamptz, NOT NULL), `event_label` (NOT NULL), `event_body`, `event_image_url`, `source_url`, `sort_order`, `metadata`, `title`, `description`, `created_at`, `updated_at`. No `type`, `content`, or `is_current` column exists in the DB. No RLS policies.

**Authoring paths.** LLM auto-generates timeline events in step 10 of the generation chain, inserted via `persist_generated_article` RPC. Admin can also add or edit events manually via StoryEditor (`StoryEditor.tsx:862`) and KidsStoryEditor (`KidsStoryEditor.tsx:701`). Both editors have a now-dead "Enrich timeline" button that calls `/api/ai/generate` — the button silently returns 400 on every click due to a parameter mismatch (`storyId`/`type` sent vs. `article_id`/`action` expected). The legacy route is already marked T69 for deletion.

**iOS adult rendering.** `StoryDetailView.swift:916` renders the Timeline tab. Fetches via `from("timelines").eq("article_id", story.id).order("event_date", ascending: true)`. Displays `event_label` + `event_body` + formatted `event_date`. "NOW" marker is fully client-side: `isCurrent` computed property always returns `nil`; the last event in the sorted array gets the badge by fallback logic (`StoryDetailView.swift:933–938`). `sort_order` is fetched but ignored — sort is `event_date ASC` only. `event_image_url` is fetched but not rendered.

**Web rendering.** None. `page.tsx` does not fetch or render timelines. `ArticleSurface.tsx` has no timeline section. Slice 03 D3 locked adding a timeline section to the web reader — the implementation spec below absorbs that decision into the new story-page architecture.

**Lenient date parse.** `parse_timeline_event_date()` (`2026-04-28_persist_timeline_lenient_date.sql:20`) handles partial dates ("2013" → Jan 1 2013, "2013-05" → May 1 2013) and returns NULL on complete parse failure. The RPC wraps it: `coalesce(parse_timeline_event_date(t->>'event_date'), now())`. A fully unparseable LLM date silently becomes `now()` — a 2013 event becomes a 2026 event with no error.

---

## The structural finding: stories need to be the top-level entity

The investigation and Q&A revealed that the flat model — one article, one timeline, one slug — doesn't match how the product should work. A running story (say, "Trump Tariffs") should be navigable as a single destination. New articles get added to it over time. The timeline is the spine connecting both plain context events and article entries. Readers land on the story and default to the most recent piece; they can navigate back through the timeline to earlier articles.

This is a container model. Stories are the top-level entity. Articles live inside stories. The timeline belongs to the story, not to an individual article.

This change is clean to make: all existing article data is test data that will be wiped. No migration of live user content is needed.

---

## Decisions

### D1 — Skip malformed-date timeline rows

**What:** Change the RPC's timeline INSERT from `coalesce(parse_timeline_event_date(...), now())` to a filtered insert that drops rows where `parse_timeline_event_date(t->>'event_date') IS NULL`. Add a `WHERE` guard:

```sql
WHERE public.parse_timeline_event_date(t->>'event_date') IS NOT NULL
```

**Why:** Silent corruption is worse than a short timeline. A 2013 event stamped as 2026 corrupts the chronological ordering and misleads readers. Dropping the row means the LLM produced something unparseable; the editorial prompt already forbids inventing dates, so this should be rare. The admin can add the event manually with a correct date if needed.

**Scope:** Update both `2026-04-28_persist_timeline_lenient_date.sql` and `2026-04-29_drop_quiz_pool_group.sql` since both contain the live RPC definition. The implementation migration supersedes both.

### D2 — Rip "Enrich timeline" button

**What:** Remove the button from `StoryEditor.tsx:956–975` and `KidsStoryEditor.tsx:790–803`.

**Why:** The button has been silently broken since it was written — parameter mismatch means every click returns 400. The route it calls (`/api/ai/generate`) is T69-marked for deletion. The AI pipeline already generates timelines at article creation time; manual editing exists for corrections. No regenerate affordance is needed speculatively.

**Scope:** Two button removals plus their surrounding error-handling blocks. The T69 route deletion happens when StoryEditor and KidsStoryEditor fully migrate to the F7 pipeline.

### D3 — Add RLS on `timelines`

**What:** Add a SELECT policy: anon and authenticated users can only read timeline rows where the parent story is published.

**Why:** Consistent with the articles RLS decision from slice 02. Without this, anyone with the anon key can read timeline events for unpublished draft stories directly via the `timelines` table.

**Policy shape (post-D5 schema):**
```sql
CREATE POLICY timelines_select_published
  ON public.timelines FOR SELECT
  TO anon, authenticated
  USING (
    EXISTS (
      SELECT 1 FROM public.stories s
      WHERE s.id = timelines.story_id
        AND s.published_at IS NOT NULL
    )
  );
```

Admin routes use service client (RLS-bypassed). iOS reads timelines only after landing on a published story — policy agrees with real traffic.

### D4 — Add `type` column to `timelines`

**What:** Add `timelines.type` column (`text`, default `'event'`, values `'event'|'article'`). When `type = 'article'`, the row also carries `timelines.linked_article_id` (uuid, nullable FK to `articles.id`).

**Why:** The story timeline has two distinct entry kinds. Plain events are context scaffolding — dates, labels, bodies. Article entries mark when a new piece was added to the story; tapping one navigates to that article. The type distinction must be persisted so both web and iOS can render the two entry kinds differently.

**Note on naming:** The existing editor UI uses `'story'` and `'event'` as type values internally (StoryEditor state). The DB column will use `'article'` and `'event'` to avoid collision with the concept of a "story" entity. The editor save path maps UI `'story'` → DB `'article'`.

**When a new article is added to a story** (via generation or re-link), the `persist_generated_article` RPC automatically inserts a `type='article'` timeline entry with `linked_article_id` pointing to the new article. Plain events generated by the LLM for that article are inserted as `type='event'` entries on the same story timeline.

### D5 — Stories as the top-level container

**What:** New `stories` table. Articles are children of stories. Timelines belong to stories, not articles. Existing test data wiped — no live data migration needed.

**Schema — `stories` table:**
```
id             uuid PK
slug           text UNIQUE NOT NULL
title          text NOT NULL
subtitle       text
description    text
published_at   timestamptz   -- NULL until first article published
created_at     timestamptz
updated_at     timestamptz
```

**Schema changes:**
- `articles`: add `story_id` uuid FK → `stories.id` (NOT NULL going forward). Remove `articles.slug` — slug lives on `stories` only.
- `timelines`: change parent FK from `article_id` to `story_id` (uuid FK → `stories.id`). Add `linked_article_id` optional FK → `articles.id` (used only when `type='article'`). Remove old `article_id` FK.
- `comments`: add `story_id` optional FK → `stories.id` for story-level discussion. Details deferred to slice 06; the column is added in the slice 05 migration as a nullable column with no constraints yet.

**What "published" means for a story:** `stories.published_at` is set when the first article within the story is published (via the PATCH route in slice 02). If all articles are archived, `published_at` is not cleared — a story remains navigable once any article was ever published. Slice 02 implementation updates this on article publish.

**Slug collision retry:** Slug is assigned at story creation. The current collision retry logic in `persist_generated_article` (appending `-2`/`-3`) migrates to the story level. Articles have no public slug.

### D6 — Generation creates story + article atomically

**What:** `persist_generated_article` RPC inserts into `stories` + `articles` + `timelines` + `quizzes` + `sources` in one transaction. The story row is created with the slug and title derived from the generated article. An `type='article'` timeline entry is automatically created pointing to the new article.

**Re-linking:** Admin can change which story an article belongs to via a story picker in StoryEditor. The picker shows existing stories (by title + slug). Selecting one updates `articles.story_id` and moves the article's `type='article'` timeline entry to the new story's timeline. Plain `type='event'` entries stay on their original story.

**Generating into an existing story:** The generation UI (Newsroom AudienceCard) will gain a "Add to existing story" option alongside the current "Generate new story" default. When selected, the RPC receives an existing `story_id` and skips creating a new stories row.

### D7 — Web reader becomes story page; iOS timeline fetch updates

**What:** `/[slug]/page.tsx` becomes a story page. The slug resolves against `stories.slug` instead of `articles.slug`. Defaults to the most recent article (`published_at DESC`). Query param `?a=<article-id>` deep-links to a specific article.

**Web story page structure:**
- Server fetches: story row, all published articles in the story (ordered by `published_at DESC`), timeline rows for the story (ordered by `event_date ASC`), sources for the default article.
- Renders: story title + description at top, then a `StoryTimeline` component showing the full event + article spine, then the selected article's body + sources + quiz + two discussion zones.
- Switching articles: clicking an article-type timeline entry pushes `?a=<article-id>` to the URL; a client component swaps article content without a full page reload. Browser back/forward works.
- Two discussion zones per view: story-level (always present, scoped to `story_id`) and article-level (for the currently selected article, scoped to `article_id`). Slice 06 handles the comment schema in full.

**Timeline rendering — web:** `StoryTimeline` component, new. Renders events and article entries in `event_date ASC` order. Event entries: date + label + body. Article entries: date + article headline, styled as a tappable card. "NOW" marker on the last entry (client-side, same logic as iOS). Graceful no-op if no timeline rows exist.

**iOS `StoryDetailView.swift` changes:**
- Timeline fetch: `eq("article_id", ...)` → `eq("story_id", ...)`.
- Article-type timeline entries (where `linked_article_id` is set) render as tappable cards; tapping loads the referenced article's data into the current view.
- Plain event entries render exactly as today.
- The Timeline tab remains always-visible with content gated by `article.view.timeline` permission (unchanged from current behavior; after slice 03 D1 RPC fix, permission resolves true for all users).

**Sitemap:** `sitemap.js` query updates from `articles` to `stories` — fetches published stories by `published_at`. One canonical URL per story slug.

## Absorbed findings (adversarial review)

- **`articles.slug` removal scope.** Seven web routes reference `articles.slug` directly (page.tsx:60, by-slug API:33, bookmarks:580, BookmarksSection:81, recap:156, admin/newsroom:423, admin/stories:155). All update to use `stories.slug` via a join or a stories-level lookup. Full enumeration is the implementation checklist item; no new owner decision needed.
- **Comment report route** constructs `/story/${article_id}` (route.js:173) — updates to `/story/${story_slug}` post-D5.
- **Dual RPC definitions.** Both `2026-04-28_persist_timeline_lenient_date.sql` and `2026-04-29_drop_quiz_pool_group.sql` contain the live RPC. The implementation migration for this slice supersedes both with a single new definition.
- **`sort_order` ignored on iOS.** iOS sorts by `event_date ASC`; `sort_order` is fetched but unused. Web will do the same. `sort_order` stays in schema for potential future manual ordering but is not a primary sort key.
- **`event_image_url` not rendered.** Present in schema, fetched by iOS, not displayed. Stays in schema; web won't render it either for now.
- **`title` and `description` columns.** Present in schema, populated by RPC if the LLM returns them, but LLM prompts don't output them. Almost always NULL in practice. Neither iOS nor web renders them; web will use `event_label` + `event_body` matching iOS.
- **D3 RLS policy** depends on `stories.published_at` existing — it does per D5 schema. Policy expression is fully specifiable without further owner input.
- **`timelines.type` DB values vs. editor UI values.** Editor state uses `'story'`/`'event'`; DB will use `'article'`/`'event'`. Editor save path maps `'story'` → `'article'`; load path maps `'article'` → `'story'`.
- **iOS parity.** iOS changes are surgical (fetch by `story_id`, tap behavior on article entries), not a full redesign. Both surfaces ship the story model together.

## Post-implementation gaps found in shipped code

These diverge from the locked plan. Flagged here so future execution sessions don't have to rediscover them.

- **`stories` table missing `subtitle` and `description` columns.** D5 schema spec included both fields. The actual migration (`2026-04-29_slice05_stories_as_containers.sql`) creates neither. Any admin UI for story-level subtitle/description requires an additional migration before it can be built.

- **`comments.story_id` FK uses `ON DELETE CASCADE`, not `ON DELETE SET NULL`.** Migration B spec said `ON DELETE SET NULL`. The actual migration uses `ON DELETE CASCADE`, which deletes all comments when a story is deleted. This is almost certainly wrong for a moderation-sensitive surface where comment history must survive content deletion. Should be corrected before launch.
