### B1. Four cron routes accept any unauthenticated caller (TODO A1)

- **Severity:** P0 if still present
- **Status:** CONFIRMED 2026-04-27 — all four still use bare `verifyCronAuth(request)` without `.ok`

### B3. `parental_consents` upsert silently fails (TODO A3)

## §C — Schedule + ops drift

### C1. `send-push` cron schedule contradiction

- **Severity:** P0 if production breaking news pushes are silently daily-batched
- **Status:** CONFIRMED 2026-04-27
- **Code comment:** `web/src/app/api/cron/send-push/route.js:21` says *"Schedule: every minute (vercel.json)"*
- **vercel.json reality:** `"schedule": "0 5 * * *"` (daily at 5am UTC)
- **Effect:** breaking-news pushes have a 24-hour delivery SLA, not minute-level. Either the comment is wrong (push really is daily) or vercel.json is wrong (push should be minute).
- **Fix:** decide which is correct. If breaking news is supposed to be near-real-time, change vercel.json to `* * * * *` or `*/2 * * * *`. If daily is intentional, fix the comment and update bible §16.

### C2. Cron schedule collisions

- **Severity:** P1 at scale
- **Status:** CONFIRMED 2026-04-27
- **Sites in `web/vercel.json`:**
  - `30 3 * * *` runs both `recompute-family-achievements` AND `anonymize-audit-log-pii`
  - `0 4 * * *` runs both `process-deletions` AND `purge-webhook-log`
- **Effect:** at modest scale, both pairs share the 60-conn Supabase pool and target overlapping tables. No symptoms today; will surface at launch.
- **Fix:** stagger by 5-15 minutes — e.g., `30 3` and `35 3`; `0 4` and `5 4`.

### C3. Two cron handlers exist but are not registered in `vercel.json`

- **Severity:** P2 (storage/perf bloat over time)

  - `web/src/app/api/cron/rate-limit-cleanup/route.ts`
- **Effect:** these run **never** unless someone hits them manually. `data-exports` storage and `rate_limit_events` table grow unbounded.
- **Fix:** either add to `vercel.json` schedule (e.g., daily at 5am for cleanup-data-exports, hourly for rate-limit-cleanup), or delete the handlers.

### C4. No `purge-error-logs` cron

- **Severity:** P2
- **Status:** INHERITED — verify
- **Effect (per R2):** `error_logs` table grows unbounded with no retention policy. Bible §17 doesn't list one.
