## CRITICAL — security holes live in production

### A1. Four cron routes accept any unauthenticated caller

**Issue.** `verifyCronAuth(request)` returns `{ok: boolean, reason?: string, vercel_cron?: boolean}`. Four routes test the object for truthiness instead of `.ok`:

- `web/src/app/api/cron/birthday-band-check/route.ts:95`
- `web/src/app/api/cron/dob-correction-cooldown/route.ts:202`
- `web/src/app/api/cron/pro-grandfather-notify/route.ts:193`
- `web/src/app/api/cron/subscription-reconcile-stripe/route.ts:163`

The returned object is always truthy, so the guard always passes. Any unauthenticated external caller can fire all four endpoints. Each runs as service-role and mutates user / billing / DOB / subscription state.

**Why fix.** Live security hole. An attacker can hit `/api/cron/subscription-reconcile-stripe` repeatedly to thrash live billing state, or `/api/cron/birthday-band-check` to force band changes on kid profiles. The other 14 crons in the codebase use `.ok` correctly — these four came from a broken template.

**Fix.** Change `if (!verifyCronAuth(request))` → `if (!verifyCronAuth(request).ok)` in all four routes. Add a unit/integration test that hits each cron with no Authorization header and asserts 403. Same four routes also have a wrong-shape `logCronHeartbeat(name, result)` call worth fixing in the same commit.

### A2. Kid pair JWT cannot pass `verifyBearerToken` (issuer mismatch)

**Issue.** `web/src/app/api/kids/pair/route.js:149` mints kid JWT with `iss: 'verity-post-kids-pair'`. `web/src/lib/auth.js:62-70` rejects any bearer whose `iss` doesn't equal `${SUPABASE_URL}/auth/v1`. Every kid-iOS authenticated call routed through `requireAuth` / `requirePermission` returns 401.

**Why fix.** The kids iOS app effectively has no working session against any permission-gated server route. The mismatch has lived long enough that both sides look settled — neither team noticed because the dev path doesn't go through `verifyBearerToken`.

**Fix.** Pick one source of truth and document. Two options:

- **(A)** Mint kid JWT with `iss = ${SUPABASE_URL}/auth/v1` and a custom `is_kid_delegated: true` claim.
- **(B)** Update `verifyBearerToken` to accept the kids-pair issuer when `decoded.is_kid_delegated === true`.

Recommend (B) — smaller blast radius, no migration of in-flight kid sessions.

### A3. parental_consents upsert silently fails (COPPA evidentiary gap)

**Issue.** `web/src/app/api/kids/pair/route.js:132` upserts with `onConflict: 'parent_user_id,kid_profile_id'`. `currentschema:1862-1874` shows `parental_consents` has only a PK on `id` — no UNIQUE constraint on `(parent_user_id, kid_profile_id)`. Postgres returns 42P10; the route swallows the error in try/catch and continues. **Consent rows are not being written for any pair-code redemption.**

**Why fix.** COPPA 16 CFR § 312.5 requires verifiable parental consent records. Today the platform cannot prove a single kid was paired with parental consent because the evidentiary table is empty. This is the row Apple's Kids Category review will ask to see.

**Fix.** Migration:
```sql
ALTER TABLE public.parental_consents
  ADD CONSTRAINT parental_consents_parent_kid_unique
  UNIQUE (parent_user_id, kid_profile_id);
```
Then verify the upsert lands a row by smoke-testing a pair flow end-to-end. Backfill missing consent rows from `kid_profiles` if the pair history is recoverable.

### A4. Apple webhook accepts Sandbox notifications in production

**Issue.** `web/src/app/api/ios/appstore/notifications/route.js` has no `notification.environment` check. The file's own comment (lines 17-18) acknowledges Sandbox and Production hit the same configured URL and the payload's `data.environment` field distinguishes — but the code never reads it.

### A5. `recordAdminAction` swallows every audit failure silently

**Issue.** `web/src/lib/adminMutation.ts:138-155` wraps the `record_admin_action` RPC in try/catch with `console.error` only — no Sentry, no fallback, no throw. If the RPC fails (auth.uid() null because cookie didn't propagate, RPC permission revoked, transient DB error), the admin mutation has already landed but the audit row hasn't.

**Why fix.** `audit_log` is the load-bearing accountability surface for every admin write. Today an unknown population of admin actions may be unaudited and nobody knows. At investigation time you query the table and find nothing — not because the action didn't happen, but because the audit write failed silently.

**Fix.** When the RPC errors:
1. Route through `captureException` with severity tag (when Sentry flips on; until then a structured log line tagged `[AUDIT-FAILURE]`).
2. Fall back to a service-role direct insert into `admin_audit_log` so the row lands even if the RPC path is broken.

### A8. 26 `DispatchQueue.main.asyncAfter` chains in kids scenes have no cancellation

## HIGH — billing / data integrity

### A9. Stripe `customer.deleted` doesn't unbind subscriptions

**Issue.** `web/src/app/api/stripe/webhook/route.js:1116-1154` clears `users.stripe_customer_id` but the matching `subscriptions` row keeps `status='active'` with the old `stripe_subscription_id`. iOS app still reads the row as live; reconciliation cron checks `stripe_subscription_id`, not customer, so the orphan never gets caught.

**Why fix.** A user whose Stripe customer was deleted (admin cleanup, Stripe-side fraud action) keeps an "active" subscription on iOS with no payment source backing it. Free service forever from the user's perspective; revenue leak from ours.

**Fix.** After clearing `stripe_customer_id`, run:
```sql
UPDATE subscriptions
   SET status='cancelled',
       cancel_reason='stripe_customer_deleted',
       updated_at=now()
 WHERE user_id=$1
   AND platform='stripe'
   AND status IN ('active','trialing');
```

### A10. account/delete leaves public.sessions rows + other-device tokens active

**Issue.** `web/src/app/api/account/delete/route.js:135-160` calls `auth.admin.deleteUser(userId)` but doesn't revoke rows in the `sessions` table. Other devices' bearer tokens stay valid until natural expiry; analytics queries see "active session" rows for deleted users.

**Why fix.** Privacy gap (a phone you forgot at the airport keeps a working session even after you deleted your account from a different device) and analytics integrity gap (deleted-user sessions inflate active counts).

**Fix.** Before `deleteUser`:
```sql
UPDATE sessions
   SET is_active=false, revoked_at=now(), revoke_reason='user_deleted'
 WHERE user_id=$1;
```
Verify `anonymize_user` RPC doesn't already do this; consolidate if it does.

### A11. promo redeem rolls back `current_uses` non-atomically

**Issue.** `web/src/app/api/promo/redeem/route.js:175-237` increments `current_uses` first, then if plan/price-resolve fails, decrements. Two concurrent redeemers can both pass the cap check before either increment is durable; if rollback fails, counter overstates uses (or understates after compensating decrements that fire twice).

**Why fix.** A capped promo (e.g., "first 100 subscribers get 50% off") can leak more than 100 redemptions. At scale this is real revenue.

### A23. Admin notifications broadcast loads ALL user IDs into memory + bulk inserts in one shot

**Issue.** `web/src/app/api/admin/notifications/broadcast/route.ts:73-100` does `service.from('users').select('id')` with no `LIMIT`, builds the full array in memory, then bulk-inserts a notification row per user in one PostgREST request. FROM users WHERE deleted_at IS NULL` server-side, capped at `LIMIT 100000`, chunked via cron-driven worker for larger broadcasts. ### A24. graduate-kid claim has no rate limit + token-existence oracle

**Issue.** `web/src/app/api/auth/graduate-kid/claim/route.ts` is a public route that takes `{token, email, password}` and calls `auth.admin.createUser`. No per-IP throttle, no per-token throttle. Returns 410 for expired tokens vs 400 for token mismatches — a token-existence oracle.

**Why fix.** Leaked or guessed tokens can be brute-forced. `auth.admin.createUser` is expensive and writes to the `auth.users` table — an attacker can both probe the token space AND create unlimited auth rows in one path. Status-code distinction tells them when they're getting warm.

**Fix.** Add `rate-limit` middleware: per-IP 10/hr + per-token 5/min. Return identical status code (e.g., 400 with generic body) for both token-mismatch and expired states. Audit the existing `rate_limit_events` plumbing to see if the helper already exists.

### A25. Auth enumeration oracle (3-route surface)

**Issue.** Three routes are individually rate-limited but combine into a complete signup-state oracle:

- `web/src/app/api/auth/check-email/route.js:38-50` returns `{available: !registered}` — direct existence signal.
- `web/src/app/api/auth/check-username/route.js:55-65` returns `{available, reserved}` — three-bit oracle.
- `web/src/app/api/auth/resolve-username/route.js` returns `{email}` for known usernames, `{email: null}` for unknown.

Combined budget: ~14.4k probes/day at per-IP cap. Enough to enumerate the entire popular-username + popular-email space.

**Why fix.** Phishing-grade signal. An attacker can build a list of all registered emails or all taken usernames and use it for credential stuffing or targeted phishing.

**Fix.** Three-part change:
1. Drop `/api/auth/check-email` — move email-uniqueness to the signup submit path with generic confirmation copy.
2. Collapse `/api/auth/check-username` to single `available` boolean (taken + reserved both → `available: false`).
3. Drop `/api/auth/resolve-username` and require email-only login, OR always return `{ok: true}` and let downstream `signInWithPassword` produce the same generic error for both wrong-password and unknown-user.

**Fix.** Drop the iOS pre-check. Trust the server trigger; on failure, parse the P0001 error and surface the cap-reached upgrade affordance. Mirrors web's pattern at the API boundary.

### A120. iOS bookmarks no delete confirmation (drift from web)
