## Q3a — Add CSAM / child-exploitation / grooming to iOS comment report sheet

Server-side routing already accepts these reasons and routes them through the priority queue (`web/src/app/api/comments/[id]/report/route.js:89-204`); this is pure client-side work.

**Why:** Federal compliance under **18 U.S.C. § 2258A** requires platforms to report CSAM. Today iOS users encountering CSAM in a comment thread can only choose "Other" with free text — which does NOT route to the urgent queue. The adult app catches a graduated 13-year-old too (kids app has no comments, but graduated kids land in the adult app), so the gap touches the kid-safety surface even though the kids app itself is comment-free. Reddit, Twitter, Substack, Discord all surface CSAM-distinguished categories on both web AND mobile.

4. **Submit handler** — Confirm the iOS report-submit POST sends the new reason strings as-is to `/api/comments/[id]/report`. Server already accepts them; no server change.
5. **Smoke test** — Mentally walk through: iOS user long-presses a comment → "Report" → picker shows 8 categories with CSAM/exploitation/grooming on top → user picks `csam` → server receives `reason='csam'` → routes to urgent queue → admin pages it. ## Q1b — Anon vs Unverified UX (banner-only)

**Decision:** **Unverified is functionally identical to Free.** No feature gates differ between unverified and verified-free. The only user-visible difference is a **banner / nudge** at the top of the authed shell with copy like "Confirm your email" or "Finish signing up." Every action a Free user can take, an Unverified user can take. Email verification becomes a soft trust signal, not a permissions wall.

**What changes in the codebase (to execute in a later session, not now):**

1. **Permissions seed** — drop the `requires_verified` column on `permissions` table OR set every existing `requires_verified=true` row to `false`. The 21 perms previously gated (comment, follow, vote, bookmark unlimited, see own activity, TTS, DM, etc.) now resolve identically for unverified and verified Free users. Migration: `UPDATE permissions SET requires_verified=false;` then drop the column in a follow-up migration once all callers are off it.
2. **`requirePermission()` resolver** — drop the `requires_verified` branch entirely. The function no longer reads `users.email_verified` for gate decisions. Reduces a code path; resolves the AUTH-MIGRATION's "unverified-with-perms" edge cases (T320, T321) by deleting the gate they were trying to carve around.
3. **AccountStateBanner** — add an `unverified` state to the tagged union. Renders nudge copy ("Confirm your email — we sent you a link" with a resend button) and a separate "Finish signing up" variant for the partial-signup case (session exists but `username IS NULL`, post-AUTH-MIGRATION). Banner is dismissible per session. Lower priority than ban/lock/freeze states (those still take precedence visually).
4. **Owner-link beta carve-out** (TODO2 T320 / T321) — **resolves automatically.** With no `requires_verified` gate anywhere, owner-link Pro recipients can comment / follow / vote / use billing settings the moment they land in the welcome flow. The `isBetaOwnerLinkSignup` bypass at `web/src/app/welcome/page.tsx:106` becomes vestigial; can be deleted as part of the AUTH-MIGRATION cleanup.
5. **Welcome carousel `isBetaOwnerLinkSignup`** — delete the boolean, delete the cohort-and-plan-id check. Welcome flow becomes uniform: every signed-in user with a session lands the same path.
6. **iOS** — no `email_verified` checks at the client side. iOS reads `requires_verified` perms today via `PermissionStore`; with the column gone, those checks evaporate.
7. **Magic-link AUTH-MIGRATION fit** — under magic-link, clicking the link IS the verification (it proves email ownership). The unverified state under banner-only model only exists in the narrow window between "session minted via password / OAuth / passwordless without email-click" — pre-AUTH-MIGRATION, this includes regular email/password signups; post-AUTH-MIGRATION, this includes only the fringe case where Supabase mints a session before the link click. Banner copy stays consistent across both regimes.
8. **"Finish signing up" partial state** — used when AUTH-MIGRATION's username-pick step (TODO2 T22) hasn't completed yet. Banner says "Finish signing up — pick a username to start posting." Tapping routes to the pick-username view.

**Why this is the right call:**
- Owner has no existing user base that depends on the verification gate; can choose the simpler product shape from a clean slate.
- Removes a maintenance class — the `requires_verified` column has historically been a source of edge cases (T320, T321 owner-link bypass, AUTH-MIGRATION carve-outs). Dropping it removes the surface entirely.
- Banner-only is faster to ship and easier to reason about. Anti-abuse signals (email-verified, account-age, quiz-passes) move to other levers (rate limits, score-tier gating, captcha-on-signup) without the perm-system dependency.
- Magic-link AUTH-MIGRATION is friendlier to banner-only — the link click already collapses signup + verification into one action; a permission gate on top is redundant.

**What stays:**
- Email verification still happens (Supabase handles the link send + token redemption). The user record still gets `email_verified=true`. We just don't gate features on it.
- Anon (no session) still hits the sign-in wall at authed routes — middleware gates `/profile/`, `/messages/`, `/notifications/`, `/bookmarks/`, etc. Unauthenticated users see the public surfaces (homepage, story, pricing, help, browse).
- Banned / locked / frozen / muted / scheduled-for-deletion states stay as functional gates with their own banner variants. Verification is the only soft state.

**Anti-abuse compensations to think about (not blocking, just flag):**
- Comment surface gets quiz-gated already (the wedge); adds a friction layer abusers have to clear.
- Score-tier requirements on certain perms (e.g., DM) can stay; verification doesn't replace them.
- Rate-limit-per-IP on signup + per-account on actions stays (existing rate-limit table).
- If anti-spam pressure surfaces post-launch, the banner-only stance can be reversed by re-adding `requires_verified` to specific perms — but the column should not come back as a default.

**Risk:** Low. Permission seed change is reversible. Banner work is small. The bigger downstream effect is that AUTH-MIGRATION's task list shrinks — T320 and T321 dissolve; T22 ("pick username post-magic-link") stays but no longer interacts with verification state.

## Q2a — Supabase Confirm-email setting + legacy cohort

**Decision:**
1. **Confirm-email setting stays DISABLED** in the Supabase Dashboard. Owner explicitly chose to leave it off.
2. **Legacy beta cohort reconciliation dissolves** — no users exist, nothing to reconcile.

**What this means in practice:**

- Supabase will mint a session **immediately** when `signInWithOtp({ email })` is called, before the user clicks the magic link. The clicked link confirms the email and stamps `users.email_verified=true`, but the session itself is live from the moment of submission.
- Under Q1b (banner-only verification, no `requires_verified` gate), this is **functionally safe** — there are no permission gates that depend on `email_verified=true`. A pre-click session can do everything a post-click session can do; the only effect of clicking the link is the banner disappearing and the verification stamp landing.
- Anti-abuse implication to flag: a fake-email signup can produce a working session that exists in DB until cleanup. Without the gate, the fake-email user can comment / vote / etc. for as long as the account stays alive. Counter-pressures: rate-limit-per-IP on signup, score-tier-gating on certain perms, the comment surface's quiz-gate (the wedge), captcha-on-signup if pressure surfaces post-launch.
- Sweep-beta cron stays in code but has nothing to operate on (zero rows). Can be left as-is or cleaned up; not blocking anything.

**Owner-side action:** None — Supabase setting stays as-is.

**Codebase action when executing AUTH-MIGRATION:** Build assuming sessions are minted on submission, not on link click. The pick-username post-signin step (Q2 item 5) runs the moment the session lands, not after the click. The `email_verified` flag is purely a soft signal for the banner's visibility — toggle banner off when `email_verified=true`.

**Why this is OK despite the recommendation:** The recommendation assumed `requires_verified` was still a gate. With Q1b dissolving the gate, the safety value of "Confirm email ON" drops sharply — it'd just be a small UX win (slightly fewer banner dismissals from spammy traffic) at the cost of a code-side complication (handling the not-yet-minted-session window in the magic-link send flow).

**Risk:** Low under Q1b. Would be Medium-High if we re-introduce a verification gate later — flag this section for re-review if `requires_verified` ever comes back.

## Q2b — Auth enumeration oracles (locked)

**Decision:** Rebuild the auth-pre-flight surface to best-practice magic-link defaults. Drop two of the three oracle routes outright, collapse the third to a session-scoped UX nicety.

**What changes in the codebase (to execute in a later session, not now):**

1. **DELETE `/api/auth/check-email`** (`web/src/app/api/auth/check-email/route.js`). No replacement. The signup form submit handler stops calling it. The form's UX becomes: user types email → submits → server responds with a generic "If that email is registered we sent you a sign-in link; otherwise we sent you a signup link. Check your inbox." regardless of whether the email exists in the DB. Same response, same status code, same latency for both cases. No oracle.
2. **DELETE `/api/auth/resolve-username`**. No replacement. Login is email-only under magic-link; mapping @handle → email has no legitimate user purpose. A user who "forgot which email" enters their best guess; the send-magic-link route silently no-ops if it doesn't match a real email and shows the same generic copy. (Reuses the no-oracle pattern from item 1.)
3. **COLLAPSE `/api/auth/check-username`** to a single boolean response shape. Returns `{available: true | false}`. Drops the `reserved` field entirely. Reserved usernames (system handles, profanity, brand-protected names) all collapse into `available: false` from the API surface — the user-facing copy on the pick-username screen says "That username isn't available — try another" with no distinction between taken-by-another-user and reserved-by-system. The endpoint becomes session-scoped: only callable by an authenticated session in the post-signin pick-username flow. Anonymous calls return 401.
4. **NEW route `/api/auth/send-magic-link`** (`web/src/app/api/auth/send-magic-link/route.js`). Single endpoint that handles both signup and signin paths. Body: `{ email }`. Calls `supabase.auth.signInWithOtp({ email, options: { shouldCreateUser: true } })`. Always returns 200 + generic body, regardless of whether the email is new (signup) or existing (signin). Replaces both the legacy login submit handler and any signup-specific email-link sender. Web `/login` and `/signup` pages both POST to this single route.
5. **Rate-limit configuration** — codify in the new `web/src/lib/rateLimits.ts` (per A129 in TODO.md). Three caps:
   - **Per-email magic-link send:** `3 / hour`, hard cap. When capped, returns the same generic "check your email" body — no rate-limit error leaks the email's existence. Internal log captures the cap event for ops visibility.
   - **Per-session username availability check:** `30 / minute`. Scoped to the authenticated session, so abusers spinning fresh sessions hit the per-IP signup cap first.
   - **Per-IP signup submit:** `5 / hour`. Combined with per-email, makes enumeration uneconomic.
6. **Status-code + response-shape hygiene** — all auth-pre-flight endpoints return 200 + the same JSON shape for both "success" and "soft-fail" cases. The only 4xx responses are for genuine input errors (malformed email, missing field). Auth-state outcomes (already-registered, locked, banned, capped) never leak through status code or response shape. Internal logs capture the actual outcome for ops.
7. **Pick-username screen wiring** — debounced inline availability check (e.g., 250ms after the user stops typing) hits `/api/auth/check-username` and renders a green check / red X next to the field. The real source of truth remains the DB UNIQUE constraint at the username-save call; the inline check is UX polish only. If the inline check returns "available" but the save then fails with a UNIQUE violation (race), the screen shows a generic "That username was taken just now — pick another."
8. **iOS callers** — `VerityPost/VerityPost/AuthViewModel.swift` updates to call `/api/auth/send-magic-link` instead of `signInWithPassword` / `auth.signUp`. The `check-username` debounce lives on the post-signin `PickUsernameView` (new screen per Q2 item 5).
9. **TODO2 items resolved by this:** T200 (signup username retry loop) and T252 (username-availability race vs `auth.signUp`) both dissolve — there's no `auth.signUp` call anymore, and the username-pick step is post-signin with the DB UNIQUE constraint as the only enforcement.

**Why this is the right call:**
- Closes a known phishing-grade enumeration surface (~14.4k probes/day per IP under current limits is enough to enumerate the popular-username + popular-email space).
- Aligns with the magic-link UX pattern adopted by Linear, Vercel, Substack, Notion — all of which return identical UX for "email exists" and "email is new" at the magic-link send step.
- Removes a maintenance class — three endpoints with subtle rate-limit interactions become one endpoint with a clean rate-limit story.
- Resolves audit items A24 and A25, plus dissolves TODO2 T200 and T252 by structural elimination.

**What stays:**
- DB UNIQUE constraint on `users.username` stays (it always was the real enforcement). No schema change.
- Magic-link emails sent by Supabase use the existing email template; copy may want a polish pass to match the new generic UX, but that's not blocking.
- The debounced inline check on the pick-username screen is a deliberate UX call — losing it would mean every user submits-and-retries which is annoying. The session-scoping + per-session rate limit makes the residual oracle surface tiny (an attacker would have to maintain ~thousands of authenticated sessions to enumerate, which costs more than it returns).

**Risk:** Low. The pre-flight endpoints are the kind of surface where deletion is safer than refactoring. Rate-limit miscalibration is the only way this hurts users — too tight a per-email cap (e.g., 1/hour) makes legitimate "email never arrived, send again" frustrating. 3/hour is a reasonable starting point; tune via ops feedback if signups stall on the cap.

## Q2d — Graduation-claim rate limit + oracle collapse (locked)

**Decision:** Apply both fixes to `web/src/app/api/auth/graduate-kid/claim/route.ts`. Add per-IP + per-token rate limits, and collapse the 410/400 status-code distinction to a single 400 with a generic body.

**What changes in the codebase (to execute in a later session, not now):**

1. **Rate-limit additions** at the top of the route handler:
   - **Per-IP:** `10 requests / hour`. Uses the existing `checkRateLimit` helper with `policyKey: 'graduate_kid_claim_ip'`.
   - **Per-token:** `5 requests / minute`. Keyed on the submitted `token` value. Prevents focused brute-force on a single guessed token.
   - Both pre-checks run before the token validation logic. On limit hit, return 429 with a generic body — but importantly, the response shape and copy match the success+invalid case enough that timing isn't an oracle on its own. (Different status code IS distinguishable, but rate-limited replies are a normal browser experience and don't leak token-existence the way 410-vs-400 does.)
2. **Status-code collapse:**
   - Both expired (was 410) and invalid (was 400) tokens return **400** with identical body: `{ error: "This signup link isn't valid. Please ask your parent for a new one." }`.
   - The mismatch-vs-expired distinction is preserved in the **internal audit log** (`audit_log` row with `action='graduate_claim_failed'` and `metadata.reason in ('token_expired','token_invalid','token_already_used')`) so ops can still debug. The audit row also captures the IP + token hash + timestamp.
3. **Response-body hygiene** — the route currently returns different copy for different failure modes (e.g., a different message when the kid_profile_id can't be resolved). Collapse all kid-claim-side failures into the same generic copy. The audit log captures the real reason. Internal-only error messages stay in server logs, never reach the response body.
4. **Add `RATE_LIMITS.AUTH_GRADUATE_CLAIM`** constants to the new `web/src/lib/rateLimits.ts` (per Q2b item 5 + audit A129 in TODO.md). Bundles cleanly with the broader rate-limit consolidation.
5. **Test path** — manually probe the route with: (a) a valid token from a real kid_profile, (b) an expired token, (c) a random invalid token, (d) the same valid token replayed past redemption. Confirm all four error paths return `{error: "This signup link isn't valid…"}` with status 400. Confirm `audit_log` rows distinguish the four cases internally.
6. **Smoke test** — run a graduation flow end-to-end on a test kid: parent generates a graduation link, kid clicks, kid completes the claim form, claim succeeds → `auth.users` row created, `kid_profiles.graduated_to_user_id` populated, `kid_profiles.status='graduated'`. Same flow with a deliberately-expired token (set `expires_at` in the past via DB) → claim fails with the new generic body, audit log shows `token_expired` reason.

**Why this matters:**
- Graduation is the highest-stakes auth surface in the product. A hijacked claim means an attacker takes over a 13-year-old's quiz history, badges, score, follows, and graduates them into an adult account under attacker control. Best-practice rate-limit + oracle-collapse is non-negotiable here.
- The token-existence oracle (410 vs 400) means an attacker probing random token strings learns when they hit a real-but-expired token. With 256-bit tokens this is theoretically uneconomic, but the principle of "no oracle on auth surfaces" applies regardless of token entropy.
- Pairs with TODO **A24** which flagged this exact issue. Closes the audit item.

**Risk:** Low. Internal audit log preserves debug-ability; user-facing copy is the only outward-visible change. Rate-limit miscalibration could affect legitimate retry attempts (a kid with a flaky connection hitting "submit" multiple times) — 10/hr per IP is loose enough to handle that case; 5/min per token is the only cap that could bite, and a real kid retrying within a minute is rare.
